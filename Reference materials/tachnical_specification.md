## 1. Общие сведения о проекте

### 1.1 Назначение системы

Система учёта студентов — веб-приложение для управления данными студентов, сотрудников и родителей в учебном заведении. Система предназначена для внутреннего использования администрацией и преподавательским составом на одном общем компьютере в учебном заведении.

Система не предназначена для работы студентов — учёт оценок и расписания намеренно исключён, поскольку для этих целей существуют специализированные платформы (МЭШ и аналоги).

### 1.2 Цели разработки

- Обеспечить централизованное хранение личных дел студентов
- Автоматизировать управление структурой учебного заведения
- Разграничить доступ по ролям пользователей
- Обеспечить безопасное удаление данных с подтверждением
- Сохранять историю всех изменений в системе

### 1.3 Универсальность

Система разработана как универсальная и может использоваться как в колледже так и в университете. Иерархическая структура факультет — группа — студент адаптируется под любое учебное заведение.

- `INSTALLED_APPS` — подключить core, crispy_forms
- `DATABASES` — настройка SQLite
- `MEDIA_ROOT` и `MEDIA_URL` — пути для загружаемых файлов

## 3. База данных

### 3.1 Описание структуры

База данных построена на иерархии: **Faculty → Group → Student**. Все таблицы используют целочисленный первичный ключ `id`. Все связи реализованы через внешние ключи (FK).

### 3.3 Связи между таблицами

| От       | К             | Тип              | Описание                                |
| -------- | ------------- | ---------------- | --------------------------------------- |
| Faculty  | Group         | Один ко многим   | Один факультет содержит много групп     |
| Group    | Student       | Один ко многим   | Одна группа содержит много студентов    |
| Faculty  | Student       | Один ко многим   | Всегда обязательна даже без группы      |
| Student  | Parent        | Многие ко многим | Через StudentParent                     |
| Position | Employee      | Один ко многим   | Одна должность у многих сотрудников     |
| Employee | Group         | Один ко многим   | Классный руководитель нескольких групп  |
| User     | AuditLog      | Один ко многим   | Один пользователь создаёт много записей |
| User     | DeleteRequest | Один ко многим   | Один пользователь создаёт много заявок  |
| User     | Employee      | Один к одному    | Один аккаунт на одного сотрудника       |

### 3.4 Каскадное удаление

- При удалении **Faculty** — удаляются все связанные **Group** и все их **Student**
- При удалении **Group** — удаляются все связанные **Student**
- Удаление выполняется **ТОЛЬКО суперадминистратором** после ввода специального пароля подтверждения
- Администраторы не могут удалять напрямую — только подавать заявку через **DeleteRequest**

## 5. Функциональные требования

### 5.1 Авторизация

- Страница входа при открытии приложения
- Поля: логин и пароль
- Без функции "запомнить меня" — каждый раз вводить заново
- После входа — редирект на дашборд в зависимости от роли
- Защита всех страниц — без авторизации доступ закрыт

### 5.2 Дашборды по ролям

- **Суперадмин** — полная статистика, список заявок на удаление, управление аккаунтами
- **Администратор** — управление студентами, группами, факультетами, сотрудниками
- **Преподаватель** — только свои группы, свои студенты, свои предметы

### 5.3 Управление структурой

- CRUD операции для Faculty
- CRUD операции для Group с привязкой к факультету и классному руководителю
- CRUD операции для Subject — список предметов
- CRUD операции для Position — только суперадмин

### 5.4 Управление студентами

- Добавление студента с заполнением всех личных данных
- Привязка студента к факультету при подаче документов
- Распределение абитуриента в группу после зачисления
- Смена статуса студента с сохранением истории в AuditLog
- Загрузка документов студента (файлы любого типа)
- Привязка родителей с указанием типа связи

### 5.5 Управление сотрудниками

- Добавление сотрудника с личными данными и должностью
- Назначение сотрудника классным руководителем группы
- Назначение сотрудника на предмет в конкретной группе
- Загрузка документов сотрудника

### 5.6 Удаление с подтверждением

- Администратор создаёт DeleteRequest с указанием причины
- Суперадмин видит список ожидающих заявок на дашборде
- Суперадмин одобряет или отклоняет каждую заявку
- При одобрении — ввод специального пароля подтверждения
- После удаления — запись в AuditLog

### 5.7 Документы

- Загрузка файлов для студентов, сотрудников и родителей
- Хранение в папке `media/` внутри проекта
- Типы: паспорт, СНИЛС, полис ОМС, аттестат, приказ, прочее
- Просмотр и скачивание загруженных документов

### 5.8 Поиск и фильтрация

- Поиск студента по ФИО
- Фильтрация по факультету
- Фильтрация по группе
- Фильтрация по статусу

### 5.9 Экспорт

- Выгрузка списка студентов группы в Excel
- Использование библиотеки openpyxl

### 5.10 Журнал изменений (AuditLog)

- Автоматическая запись всех изменений данных
- Фиксируется: кто, что, когда изменил
- Хранится состояние объекта до и после изменения в формате JSON
- Просмотр журнала доступен только суперадмину

## 6. Нефункциональные требования

### 6.1 Интерфейс

- Веб-приложение — работает в браузере
- Дизайн на Bootstrap 5 — стандартный, минималистичный
- Дизайн реализуется в последнюю очередь после стабильной работы функционала
- Адаптивность не является приоритетом — работа на десктопе

### 6.2 Безопасность

- Все пароли хранятся в хэшированном виде — Django управляет автоматически
- Все страницы защищены декораторами авторизации
- Разграничение доступа по ролям через middleware или декораторы
- Специальный пароль подтверждения для критических операций удаления

### 6.3 Хранение файлов

- Файлы хранятся в папке `media/` внутри проекта
- Папка `media/` не включается в Git репозиторий
- Пути к файлам хранятся в БД как строки

### 6.4 Версионирование

- Git репозиторий на GitHub
- Коммит после каждой выполненной задачи
- Понятные сообщения коммитов на русском или английском

```bash
# 2. Создать виртуальное окружение
python -m venv venv

# 3. Активировать
venv\Scripts\activate

# 4. Установить зависимости
pip install -r requirements.txt

# 5. Применить миграции
python manage.py migrate

# 6. Создать суперадмина
python manage.py createsuperuser

# 7. Запустить сервер
python manage.py runserver
```

## 10. Соглашения по коду

### 10.1 Именование

- Модели — `CamelCase` (Student, GroupSubjectEmployee)
- Поля моделей — `snake_case` (first_name, group_id)
- URL маршруты — `kebab-case` (student-list, add-student)
- Функции и переменные Python — `snake_case`
- Шаблоны — `snake_case` с расширением .html (student_list.html)

### 10.2 Сообщения коммитов Git

| Префикс     | Назначение                                 |
| ----------- | ------------------------------------------ |
| `feat:`     | Добавить новую функцию                     |
| `fix:`      | Исправить ошибку                           |
| `refactor:` | Переработать код без изменения функционала |
| `docs:`     | Обновить документацию                      |
| `style:`    | Изменения интерфейса                       |
| `db:`       | Изменения в моделях или миграциях          |

### 10.3 Язык кода

- Код и комментарии — английский
- Сообщения коммитов — русский или английский
- Интерфейс пользователя — русский

## 11. Глоссарий

| Термин           | Определение                                                       |
| ---------------- | ----------------------------------------------------------------- |
| Django           | Python веб-фреймворк для быстрой разработки веб-приложений        |
| SQLite           | Встроенная реляционная база данных, хранится в одном файле        |
| Model            | Класс Python описывающий таблицу в базе данных                    |
| Migration        | Файл с инструкциями для изменения структуры БД                    |
| View             | Функция или класс обрабатывающий HTTP запрос и возвращающий ответ |
| Template         | HTML файл с шаблонными тегами Django для динамического контента   |
| URL              | Маршрут — адрес страницы в приложении                             |
| FK (Foreign Key) | Внешний ключ — ссылка на запись в другой таблице                  |
| PK (Primary Key) | Первичный ключ — уникальный идентификатор записи                  |
| CRUD             | Create Read Update Delete — базовые операции с данными            |
| Bootstrap        | CSS фреймворк для быстрой вёрстки интерфейса                      |
| Media            | Папка для хранения загружаемых пользователями файлов              |
| Virtual env      | Изолированное Python окружение для проекта                        |
| AuditLog         | Журнал автоматической фиксации всех изменений в системе           |
| DeleteRequest    | Заявка администратора на удаление объекта                         |
| choices          | Фиксированный список допустимых значений для поля в Django        |
| ImageField       | Поле Django для хранения изображений                              |
| FileField        | Поле Django для хранения файлов любого типа                       |
| Nullable         | Поле которое может быть пустым (null=True в Django)               |
