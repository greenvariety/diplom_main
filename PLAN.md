## ~~Задача 0: Инфраструктура — DRF + React + JWT~~

### О прототипе
В папке `Прототип/` в корне проекта лежит готовый React-прототип всего сайта. Его нужно взять и использовать как основу фронтенда — не переписывать и не переводить в другой синтаксис, а использовать напрямую. Файлы прототипа:
- `styles.css` — вся дизайн-система (CSS-переменные, все компоненты)
- `shell.jsx` — навигационная оболочка приложения (sidebar + topbar)
- `screens.jsx` — все экраны приложения
- `modals.jsx` — все модальные формы
- `auth.jsx` — экраны авторизации (логин, регистрация, восстановление)
- `data.jsx` — SVG-иконки и структуры демо-данных
- `utils.jsx` — переиспользуемые UI-компоненты
- `tweaks-panel.jsx` — панель настройки темы

Архитектура: React SPA (фронтенд) + Django REST Framework (бэкенд/API). Django больше не рендерит HTML — только отдаёт данные через `/api/`. Фронтенд живёт в папке `frontend/` и работает на Vite.

### Контекст проекта
Django-проект (АИС учёт студентов). Весь код в приложении `core`. Модели в `core/models.py`, настройки в `config/settings.py`. Это **первая задача** — подготовить всю инфраструктуру до того, как трогать конкретные страницы.

### Что сделать

**1. Установить Python-пакеты** (добавить в `requirements.txt` и установить):
- `djangorestframework`
- `djangorestframework-simplejwt`
- `django-cors-headers`

**2. Настроить `config/settings.py`**:
- Добавить в `INSTALLED_APPS`: `rest_framework`, `corsheaders`
- Добавить `corsheaders.middleware.CorsMiddleware` первым в `MIDDLEWARE`
- Добавить настройки:
```python
CORS_ALLOWED_ORIGINS = ["http://localhost:5173"]
REST_FRAMEWORK = {
    'DEFAULT_AUTHENTICATION_CLASSES': ['rest_framework_simplejwt.authentication.JWTAuthentication'],
    'DEFAULT_PERMISSION_CLASSES': ['rest_framework.permissions.IsAuthenticated'],
}
from datetime import timedelta
SIMPLE_JWT = {
    'ACCESS_TOKEN_LIFETIME': timedelta(minutes=60),
    'REFRESH_TOKEN_LIFETIME': timedelta(days=30),
}
```

**3. Создать папку `frontend/`** с Vite + React (plain JavaScript, без TypeScript):
```
npm create vite@latest frontend -- --template react
cd frontend
npm install
npm install axios
```

**4. Скопировать прототип в frontend**:
- `Прототип/styles.css` → `frontend/src/styles.css`
- `Прототип/shell.jsx` → `frontend/src/shell.jsx`
- `Прототип/screens.jsx` → `frontend/src/screens.jsx`
- `Прототип/modals.jsx` → `frontend/src/modals.jsx`
- `Прототип/auth.jsx` → `frontend/src/auth.jsx`
- `Прототип/data.jsx` → `frontend/src/data.jsx`
- `Прототип/utils.jsx` → `frontend/src/utils.jsx`
- `Прототип/tweaks-panel.jsx` → `frontend/src/tweaks-panel.jsx`

**5. Создать `frontend/src/api.js`** — axios-клиент с JWT:
- Базовый URL: `http://localhost:8000/api`
- Автоматически добавляет `Authorization: Bearer <access_token>` из `localStorage`
- При 401 — автоматически запрашивает новый access-токен через `/api/auth/refresh/`
- Если refresh тоже истёк — редиректит на логин

**6. Настроить `frontend/vite.config.js`** — proxy на Django:
```js
server: {
  proxy: {
    '/api': 'http://127.0.0.1:8000',
    '/media': 'http://127.0.0.1:8000',
  }
}
```

**7. Обновить `frontend/src/main.jsx`** — точка входа (пока просто рендерит заглушку "Готово")

**8. В `core/urls.py`** добавить базовый роут `/api/` (пустой DRF router — будет заполняться в следующих задачах)

### Проверка
- `cd frontend && npm run dev` — Vite запускается на `:5173`
- `python manage.py runserver` — Django запускается на `:8000`
- Страница `http://localhost:5173` открывается без ошибок в консоли

### О прототипе
В папке `Прототип/` в корне проекта лежит готовый React-прототип всего сайта. Его нужно взять и использовать как основу фронтенда — не переписывать и не переводить в другой синтаксис, а использовать напрямую. Файлы прототипа:

Архитектура: React SPA (фронтенд) + Django REST Framework (бэкенд/API). Django больше не рендерит HTML — только отдаёт данные через `/api/`. Фронтенд живёт в папке `frontend/` и работает на Vite. Инфраструктура уже настроена (Задача 0 выполнена).

### Контекст проекта
Django-проект (АИС учёт студентов). Модели в `core/models.py`: `User` (кастомная с полем `role`: superadmin/admin/teacher), `Institution` (организация), `SeedPhrase` (набор слов для восстановления). Существующая логика регистрации — в `core/views.py` (функции `register_view`, `recover_password_view`) — можно брать за образец бизнес-логики.

### Что сделать

**1. Взять `Прототип/auth.jsx`** — он содержит 4 готовых экрана: `LoginScreen`, `RegisterScreen`, `SeedPhraseScreen`, `RecoverScreen`. Сейчас они используют mock-данные и `setScreen()` для навигации. Нужно заменить на реальные API-вызовы.

**2. Создать `core/api_auth.py`** с DRF-вьюхами:
- `POST /api/auth/login/` — принимает `{email, password}`, возвращает `{access, refresh, user: {id, email, role, full_name}}`
- `POST /api/auth/register/` — принимает `{org_name, email, password}`, создаёт Institution + User(superadmin) + генерирует seed-фразу (12 случайных слов), возвращает `{access, refresh, seed_phrase: ["слово1", ...]}`
- `POST /api/auth/refresh/` — принимает `{refresh}`, возвращает `{access}`
- `POST /api/auth/logout/` — возвращает 200 (токен удаляется на клиенте)
- `POST /api/auth/recover/` — принимает `{seed_words: [...12 слов...], new_password}`, находит пользователя по совпадению seed-фразы, меняет пароль, возвращает `{access, refresh}`

**3. Добавить роуты в `core/urls.py`**

**4. Обновить `frontend/src/auth.jsx`**:
- `LoginScreen`: при сабмите — `POST /api/auth/login/`, сохранить токены в `localStorage`, перейти в приложение
- `RegisterScreen`: при сабмите — `POST /api/auth/register/`, сохранить токены, перейти на `SeedPhraseScreen` передав seed-слова
- `SeedPhraseScreen`: только отображение полученных seed-слов, кнопка "Продолжить" → в приложение
- `RecoverScreen`: при сабмите — `POST /api/auth/recover/`, сохранить токены → в приложение
- Показывать ошибки API в форме

**5. Обновить `frontend/src/main.jsx`**:
- Если в `localStorage` нет `access_token` → рендерить `auth.jsx` (`LoginScreen`)
- Если токен есть → рендерить основное приложение (пока заглушка — будет в Задаче 2)

### Проверка
- Регистрация → показывает seed-фразу → попадает в приложение
- Логин с верными данными → попадает в приложение
- Восстановление по seed-словам → попадает в приложение
- При неверных данных — показывается ошибка в форме
