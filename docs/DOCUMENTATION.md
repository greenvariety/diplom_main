## 1. Общее описание системы

**АИСК** — Автоматизированная Информационная Система Колледжа. Веб-приложение для управления учебным заведением: студентами, сотрудниками, группами, факультетами, предметами и документами.

### Для кого предназначена

- **Владелец** (директор/ИТ-администратор) — настройка системы, управление организацией, контроль всех операций.
- **Администратор** (заместитель директора, завуч) — управление структурой учебного заведения и персоналом.
- **Секретарь** — ввод и ведение данных о студентах и опекунах.
- **Преподаватель** — просмотр информации по своим группам и студентам.

### Задачи, которые решает система

- Ведение реестра студентов с историей статусов (поступление, обучение, отчисление, перевод).
- Управление структурой учебного заведения: факультеты, группы, должности.
- Хранение данных сотрудников и прикреплённых к ним учебных нагрузок.
- Хранение документов (паспорт, СНИЛС, аттестат и др.) для студентов, сотрудников и опекунов.
- Фиксация всех изменений данных в журнале аудита с возможностью отката.
- Мультитенантность: один владелец может управлять несколькими организациями.
- Двухуровневый процесс удаления: администратор подаёт заявку, владелец одобряет.
- Email-подтверждение при регистрации и восстановлении пароля.

```

  │

  ▼

  │
  │  React SPA загружается в браузере
  │
  │  POST /api/auth/login/
  │  GET  /api/me/
  │  GET  /api/students/?page=1
  │  ...  (все /api/... запросы)
  ▼
Django REST Framework (DRF APIView)
  │
  │  JWT-токен в заголовке Authorization: Bearer <token>

  ▼
SQLite (db.sqlite3)
```

**Медиафайлы** (`/media/`) обслуживаются Django напрямую в режиме разработки.

## 3. База данных — модели

Все модели определены в `core/models.py`. Используется SQLite. Кастомный пользователь наследует `AbstractBaseUser`.

### Institution — Учебное заведение

Корневая сущность системы. Все остальные объекты привязаны к `Institution`.

| Поле | Тип | Описание |
|---|---|---|
| `id` | AutoField | Первичный ключ |
| `owner` | FK → User | Владелец (пользователь с ролью owner) |
| `code` | CharField(50) | Аббревиатура / короткий код (уникален в пределах owner) |
| `name` | CharField(1000) | Полное название |
| `description` | TextField(5000) | Описание |
| `photo` | ImageField | Фото / логотип |
| `founded_date` | DateField | Дата основания |
| `created_at` | DateTimeField | Дата добавления в систему (авто) |

Уникальность: `(owner, code)`.

## 4. API — эндпоинты

Все эндпоинты начинаются с `/api/`. Для аутентификации используется JWT-токен в заголовке:

```
Authorization: Bearer <access_token>
```

Исключение — публичные эндпоинты аутентификации (`AllowAny`).

**Коды ответов:**
- `200 OK` — успех
- `201 Created` — объект создан
- `204 No Content` — объект удалён (без тела ответа)
- `400 Bad Request` — ошибка валидации
- `401 Unauthorized` — токен отсутствует или недействителен
- `403 Forbidden` — недостаточно прав
- `404 Not Found` — объект не найден
- `429 Too Many Requests` — превышен лимит запросов

---

### 4.1. Аутентификация (`api_auth.py`)

#### POST `/api/auth/login/`
Вход в систему. Публичный.

**Тело запроса:**
```json
{ "username": "ivanov", "password": "pass1234" }
```

**Ответ:**
```json
{
  "access": "<jwt_access_token>",
  "refresh": "<jwt_refresh_token>",
  "user": {
    "id": 1,
    "username": "ivanov",
    "role": "teacher",
    "full_name": "Иванов Иван Иванович",
    "institution": { "id": 2, "name": "Тестовый колледж", "code": "ТК" }
  }
}
```

---

#### POST `/api/auth/register/`
Первый шаг регистрации владельца. Отправляет код на email. Публичный.

**Тело запроса:**
```json
{ "login": "petrov", "name": "Петров Пётр", "email": "p@mail.ru", "pass": "Secure1!" }
```

**Ответ:** `{ "masked_email": "p***@mail.ru" }`

---

#### POST `/api/auth/verify-email/`
Второй шаг регистрации. Подтверждает email-код и создаёт аккаунт владельца. Публичный.

**Тело запроса:**
```json
{ "login": "petrov", "code": "123456" }
```

**Ответ:** аналогичен `/login/` — возвращает `access`, `refresh`, `user`.

---

#### POST `/api/auth/resend-register-code/`
Повторная отправка кода регистрации. Публичный.

**Тело запроса:** `{ "login": "petrov" }`

---

#### POST `/api/auth/refresh/`
Обновление access-токена. Публичный.

**Тело запроса:** `{ "refresh": "<refresh_token>" }`

---

#### POST `/api/auth/logout/`
Выход из системы. Требует токен.

#### POST `/api/auth/recover/send-code/`

## 5. Модули системы

### 5.1. Аутентификация и регистрация

Регистрация доступна только для роли `owner` (руководитель учебного заведения). Процесс двухшаговый:
1. Пользователь заполняет форму (логин, ФИО, email, пароль) — отправляется код на email.
2. Вводит 6-значный код → аккаунт создаётся, пользователь автоматически входит.

Защита от перебора: максимум 5 кодов за 15 минут, 5 попыток на код.

Восстановление пароля работает только для роли `owner` через email.

JWT access-токен живёт 60 минут, refresh-токен — 30 дней.

---

### 5.2. Мультитенантность

Один владелец может создать несколько организаций. Пользователи (`admin`/`teacher`/`secretary`) привязываются к одной организации при создании. Владелец может переключаться между своими организациями — все данные фильтруются по `institution`.

При переключении организации (`/api/organizations/<id>/switch/`) обновляется поле `user.institution` и все последующие API-запросы работают в контексте новой организации.

---

### 5.3. Управление структурой

**Факультеты** — структурные подразделения. Код факультета (аббревиатура) генерируется автоматически из первых букв слов названия.

**Группы** — привязаны к факультету. Номер группы автоматически рассчитывается при создании. Название формируется по формуле `<ФакShort>-<Номер>-<ГодКороткий>`.

**Должности** — определяют роль сотрудника в системе (`teacher`, `admin`, `secretary`, `none`).

**Предметы** — назначаются группам через таблицу `GroupSubjectEmployee` с указанием преподавателя.

---

### 5.4. Работа со студентами

Студент имеет статус с чётким жизненным циклом (см. раздел 3, модель `Student`). Статус меняется вручную через PATCH-запрос.

К студенту можно привязать несколько опекунов с указанием типа связи (мать, отец, опекун).

Студента можно перевести в другую группу внутри организации или в другую организацию того же владельца.

---

### 5.5. Документы

Документы хранятся в папке `media/documents/` и привязываются к физическим лицам (студент, сотрудник, опекун) через поля `owner_type` / `owner_id`.

Поддерживаются типы: паспорт, СНИЛС, полис ОМС, аттестат, приказ, прочее.

Изображения (jpg, jpeg, png, gif, webp) имеют флаг `is_image = True` — фронтенд показывает их как превью.

---

### 5.6. Двухуровневое удаление

Объекты не удаляются напрямую пользователями с ролью `admin`/`secretary`. Вместо этого:
1. `admin`/`secretary` создаёт `DeleteRequest` с указанием причины.
2. `owner` видит все заявки в разделе «Заявки на удаление» и может одобрить (объект удаляется) или отклонить заявку.
3. Владелец может удалять объекты напрямую (с подтверждением паролем).

Исключения: удаление студентов, родителей — доступно `owner`/`admin`/`secretary` напрямую с вводом пароля.

---

### 5.7. Журнал изменений (аудит)

Все CRUD-операции фиксируются в `AuditLog` через вызов `log_action()` в `utils.py`. В журнале хранятся данные до и после изменения в виде JSON.

Функции журнала:
- Фильтрация по пользователю, типу объекта, периоду, действию.
- Экспорт в Excel (с детализацией по изменённым полям).
- Откат изменения: для записей типа `updated` можно восстановить предыдущие значения полей.
- Переход к объекту: из записи журнала можно перейти к странице студента/сотрудника/группы.

Администратор не видит действий владельца в журнале.

---

### 5.8. Вопросы к записям (RecordNote)

`admin`/`owner` могут добавить вопрос/пометку к любой записи студента, сотрудника или группы. Вопрос отображается в списке синей иконкой `?`. Только владелец может закрыть вопрос.

### 5.9. Фронтенд (React SPA)

Приложение — одностраничное (SPA). Все экраны рендерятся на стороне браузера.

**Основные файлы:**

| Файл | Назначение |
|---|---|
| `App.jsx` | Корневой компонент, управление состоянием (токен, текущий пользователь, навигация) |
| `shell.jsx` | Оболочка после авторизации: сайдбар, топбар, переключатель организаций |
| `auth.jsx` | Экраны: вход, регистрация, выбор организации, восстановление пароля |
| `screens.jsx` | Все экраны-списки и детальные страницы (студенты, сотрудники, группы и т.д.) |
| `modals.jsx` | Все модальные окна (формы создания/редактирования) |
| `profile.jsx` | Страница профиля пользователя (три вкладки: данные, пароль, email) |
| `api.js` | Все вызовы к REST API (fetch + Bearer-токен) |

**Сайдбар** показывает разные пункты меню в зависимости от роли:
- `owner`: все разделы включая Пользователи, Журнал изменений, Заявки на удаление.
- `admin`: Студенты, Сотрудники, Группы, Факультеты, Предметы, Журнал, Заявки.
- `secretary`: Студенты, Опекуны, Сотрудники.
- `teacher`: только Студенты и Группы (только свои).

## 7. Установка и запуск

### Требования

- Python 3.10+
- Node.js 18+

### Шаг 1 — Клонировать репозиторий

```bash
git clone <url-репозитория>
cd <папка-проекта>
```

### Шаг 2 — Создать виртуальное окружение и установить зависимости

```bash
python -m venv venv
venv\Scripts\activate          # Windows
# source venv/bin/activate     # Linux/Mac

pip install -r requirements.txt
```

### Шаг 3 — Настроить переменные окружения

Создать файл `.env` в корне проекта (рядом с `manage.py`):

```env
SECRET_KEY=your-secret-key-here
EMAIL_HOST_USER=your-gmail@gmail.com
EMAIL_HOST_PASSWORD=your-app-password-16-chars
```

- `SECRET_KEY` — произвольная строка для шифрования сессий.
- `EMAIL_HOST_USER` и `EMAIL_HOST_PASSWORD` — Gmail-аккаунт и App Password (16 символов без пробелов из настроек безопасности Google).

### Шаг 4 — Применить миграции базы данных

```bash
python manage.py migrate
```

### Шаг 5 — Создать суперадминистратора

```bash
python manage.py create_superadmin
```

Команда создаёт пользователя с ролью `owner`. Следуйте инструкциям в терминале.

### Шаг 6 (опционально) — Заполнить тестовыми данными

```bash
python manage.py seed_data
```

Создаёт организацию с факультетами, группами, студентами, сотрудниками и предметами.

### Шаг 7 — Запустить бэкенд

```bash
python manage.py runserver
```

Сервер запускается на `http://127.0.0.1:8000/`.

### Шаг 8 — Сборка фронтенда (при изменении исходников)

```bash
cd frontend
npm install
npm run build
```

Собранный бандл попадает в `frontend/dist/` и автоматически отдаётся Django.

### Быстрый старт (Windows)

Запустить `Запустить проект.bat` в корне репозитория — активирует окружение и запускает сервер.

### Разработка фронтенда с hot-reload

```bash
cd frontend
npm run dev
```

Dev-сервер Vite запускается на `http://localhost:5173/`. В `config/settings.py` уже настроен CORS для этого адреса.
