# Бизнес-правила

## Роли и доступ

```
owner > admin > secretary > teacher
```

| Действие | teacher | secretary | admin | owner |
|---|:---:|:---:|:---:|:---:|
| Просмотр своих групп/студентов | ✓ | ✓ | ✓ | ✓ |
| Просмотр всех групп/студентов | — | ✓ | ✓ | ✓ |
| Создание/редактирование сущностей | — | ✓ | ✓ | ✓ |
| Управление пользователями | — | — | — | ✓ |
| Управление должностями | — | — | — | ✓ |
| Удаление (прямое) | — | — | — | ✓ |
| Создание заявки на удаление | — | ✓ | ✓ | — |
| Одобрение/отклонение заявок | — | — | — | ✓ |
| Просмотр журнала изменений | — | — | ✓ | ✓ |
| Добавление вопроса к записи (RecordNote) | — | ✓ | ✓ | ✓ |
| Закрытие вопроса к записи | — | — | — | ✓ |
| Создание/управление организациями | — | — | — | ✓ |

Реализация: `request.user.is_owner`, `request.user.is_admin`, `request.user.is_secretary`, `request.user.is_teacher_role` в DRF-вьюхах. Декораторы `@owner_required` / `@admin_required` в `core/utils.py` (только для Django-вьюх).

Роль `admin` в аудит-логе: видит записи всех пользователей кроме owner.

## Должности и роли (`Position.role_type`)

Каждая должность имеет поле `role_type`, определяющее системный доступ при создании аккаунта сотруднику:
- `admin` — уровень администратора
- `secretary` — уровень секретаря  
- `teacher` — уровень преподавателя
- `none` — без системного доступа (должность есть, аккаунта нет)

При создании аккаунта сотруднику (`EmployeeAccountView`) роль берётся из `position.role_type`.

## Флаги (`is_flagged`)

Поле `is_flagged = True/False` есть у Faculty, Group, Employee, Student, Parent. Флаг ставится через отдельный endpoint `/api/<сущность>/<pk>/flag/`. Используется как визуальная метка (например, «обратить внимание»). Логика флага полностью кастомная — никакой блокировки не вызывает.

## Регистрация и роль owner

Пользователь с ролью `owner` создаётся через публичную регистрацию с обязательным подтверждением email:
1. POST `/api/auth/register/` — отправляет 6-символьный код на email.
2. POST `/api/auth/verify-email/` — проверяет код, создаёт аккаунт, возвращает JWT.

Пользователи `admin` и `teacher` создаются owner'ом вручную через `/api/users/`.

## Организации (Institution) — мультитенантность

`owner` создаёт одну или несколько `Institution`. Все данные (факультеты, группы, студенты и т.д.) принадлежат конкретной организации.

- После входа owner выбирает, с какой организацией работать (`/api/organizations/<pk>/switch/`).
- `admin` и `teacher` привязаны к конкретной организации через `user.institution`.
- Удаление организации требует подтверждения кодом из email.

## Процесс удаления объектов

**Администратор** не может удалять напрямую. Он создаёт `DeleteRequest`:
- Указывает причину удаления.
- Заявка уходит в статус `pending`.

**Владелец (owner)** рассматривает заявки (`/api/delete-requests/`):
- **Одобрить**: вводит свой пароль → объект удаляется физически → в AuditLog пишется `deleted`.
- **Отклонить**: статус меняется на `rejected`, объект остаётся.

Owner может также удалять **напрямую** через DELETE `/api/<сущность>/<id>/`.

## Статусы студента

```
pending_review → pending_enrollment → enrolled
                                         ↓
                               pending_expulsion → expelled
                                         ↓
                                      transferred
```

| Код | Название | Смысл |
|---|---|---|
| pending_review | На рассмотрении | Только подал документы |
| pending_enrollment | На зачисление | Документы проверены, готов к зачислению |
| enrolled | Зачислен | Активный студент |
| pending_expulsion | На отчисление | Инициирован процесс отчисления |
| expelled | Отчислен | Отчислен |
| transferred | Переведён | Переведён в другую группу/факультет |

Статус меняется вручную администратором через форму редактирования студента.

## Группы

- Номер группы (`group_number`) **автогенерируется** при создании: берётся количество групп на факультете + 1.
- Имя группы — вычисляемое свойство: `{faculty.short_name}-{group_number}-{year[-2:]}`.
- Один предмет в группе — только один преподаватель (`unique_together = [('group', 'subject')]`).
- Студент может быть без группы (`group = NULL`), но факультет обязателен.

## Преподаватель видит

Только группы, где он:
1. Является классным руководителем (`headteacher`), **или**
2. Ведёт хотя бы один предмет (`GroupSubjectEmployee.employee`).

Соответственно, видит только студентов этих групп.

## Документы

- Документ принадлежит владельцу через `owner_type` + `owner_id` (не `GenericForeignKey`).
- При удалении документа — физически удаляется файл с диска (`doc.file.delete(save=False)`).
- Изображения (`jpg/jpeg/png/gif/webp`) показываются inline на странице документа.

## Вопросы к записям (RecordNote)

Механизм для отметки спорных или требующих внимания записей:
- Admin/secretary/owner добавляет вопрос через `POST /api/notes/` с `object_type` и `object_id`.
- На одну запись может быть только **один** активный (незакрытый) вопрос.
- Owner закрывает вопрос через `POST /api/notes/<pk>/resolve/`.
- Создатель или owner может удалить вопрос через `DELETE /api/notes/<pk>/`.

## Уникальность email и телефона

Email и телефон проверяются на уникальность **глобально** (среди Employee, Student, Parent) через хелперы в `utils.py`:
- `check_person_email_unique(email, ...)` — проверяет по всем Employee, Student, Parent, User
- `check_person_phone_unique(phone, ...)` — проверяет по всем Employee, Student, Parent

## Смена пароля (ProfileChangePasswordView)

Правила нового пароля: минимум 8 символов, хотя бы одна цифра, хотя бы одна латинская буква, хотя бы один спецсимвол (`_-!@#$%^&*+.,;:?`). Нельзя использовать старый пароль. После успешной смены выдаётся новый JWT (access + refresh).

## Смена email (ProfileChangeEmailView / ProfileConfirmEmailView)

Смена email через двухшаговый процесс: запрос кода (`change_email` purpose в EmailCode) → подтверждение кода. Не более 3 запросов за 15 минут. Cooldown 60 секунд между повторными отправками. Максимум 5 неверных попыток ввода кода.

## Аудит

Вызов `log_action()` обязателен при любом create/update/delete.  
Данные до/после хранятся как JSON-строки в полях `old_data` / `new_data`.  
AuditLog привязан к `institution`, поэтому журнал фильтруется по текущей организации.

Откат (rollback) доступен для записей с `action='updated'` через `POST /api/audit-log/<pk>/rollback/`. Поддерживаемые типы: Student, Employee, Faculty, Group, Parent. Доступен owner и admin.
