## Сессия 2026-05-23 (4) - Глобальный автотест

Написан скрипт `tests/autotest_global.py` — комплексный авто-тест всей системы. Результат: **83/83 PASS**.

Что покрывает:
- Авторизация: вход, 401 без токена, 401 при неверном пароле, /me/, /dashboard/
- Организации: создание, список, переключение, обновление
- Должности: CRUD
- Факультеты: создание, список, обновление (у FacultyDetailView нет GET — только PATCH)
- Предметы: создание, список, обновление
- Сотрудники: CRUD, детальная страница
- Группы: создание, список, детали, назначение предмета+преподавателя
- Студенты: CRUD, смена статуса, перевод в другую группу, поиск, фильтр
- Опекуны: CRUD, привязка к студенту
- Документы: загрузка файла, GET, DELETE
- Пользователи: создание admin и teacher, список, PATCH, смена пароля, DELETE
- Права доступа: 401 без токена, 403 для teacher (создание), admin может позиции/audit-log/delete-requests
- Заявки на удаление: admin создаёт, owner видит счётчик >= 1, owner отклоняет
- Журнал изменений: GET список (26+ записей), GET users

Тестовый пользователь `autotest_owner` создаётся и удаляется автоматически через Django shell. Данные полностью очищаются после теста.

Находки при написании (не баги, а особенности реализации):
- `birth_date` и `created_at` нельзя передавать строками через API — Django хранит строку в памяти после `create()`, а `_employee_data`/`_faculty_data` вызывают `.isoformat()`/`.strftime()` на строке → 500. Фронт обходит это, не заполняя даты при создании.
- Прямого DELETE для студентов, групп, сотрудников, факультетов нет — только через delete-request approve.
- Admin может: читать audit-log, создавать должности, читать/создавать delete-requests.
- FacultyDetailView, GroupSubjectsView, StudentParentsView не имеют GET-метода.

## Сессия 2026-05-22 (2) - Бан по email при исчерпании кодов

### Блокировка по email (не по логину)

- Лимит кодов теперь считается по адресу **email**, а не по логину. Нельзя обойти блокировку, перезаполнив форму регистрации с тем же email но другим логином.
- Функция `_register_rate_check` принимает email вместо логина, фильтрует по `email=email`.

### Бан при 5-й неверной попытке на 5-м коде

- Если пользователь запросил **5 кодов** (максимум) и на последнем коде ввёл неверный код **5 раз**, то на 5-й неверной попытке получает ответ 429 с сообщением "Запросы истекли. Повторите через 15 минут" вместо "Запросите новый код" (нового кода всё равно не будет).
- То же самое, если 5 попыток уже исчерпаны и пользователь снова заходит на экран ввода последнего кода.

### Фронтенд

- При получении 429 в обработчике `submit` (ввод кода) - кнопка "Отправить снова" блокируется (isBlocked = true), точно так же как при 429 при повторной отправке.

## Сессия 2026-05-22 - Ограничение попыток кода + проверка занятости логина/email

### Ограничение попыток ввода кода при регистрации

- Теперь на каждый присланный код дается ровно **5 попыток** ввода. Если все 5 неверные - код "умирает" и под полем ввода появляется сообщение "Превышено число попыток. Запросите новый код".
- Всего можно запросить **5 кодов** (первый + 4 повторных). Если все 5 кодов исчерпаны, при попытке запросить новый - под полем ввода появляется "Запросы истекли. Повторите через 15 минут", кнопка "Отправить снова" блокируется.
- Окно блокировки - **15 минут** (было 60 минут).

### Проверка занятости логина и email при регистрации

- При потере фокуса с поля **Логин** - система проверяет, занят ли логин, и сразу под полем показывает ошибку "Логин уже занят. Выберите другой".
- При потере фокуса с поля **Email** - аналогичная проверка: "Этот email уже зарегистрирован".
- Проверка срабатывает только если введенное значение прошло локальную валидацию (формат правильный). Если пользователь меняет значение - ошибка сбрасывается.
- При сабмите формы ошибки занятости логина/email тоже отображаются под полями (а не только во всплывашке).
- Добавлена проверка уникальности email на бэкенде в RegisterView.
